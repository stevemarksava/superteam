# Game Tape — 2026-06-08 (Entry 3)
**Session:** Skills.sh over-install — Steve catches it, clean correction
**Coach trigger:** Fired on the correction push. Third game tape of the day, same day the after-push rule went live. The drill is now three-for-three.

---

## What Happened

Superman started mass-installing skills from skills.sh — Frontend, React, Next.js, Design, Mobile, Testing, Databases, Marketing, effectively the whole anthropics/skills shelf. Steve stopped it cold: *"I don't think we need to install all the skills... just the topics so you know about it. Main mission is always save context, don't waste it."*

The correction:
1. Eleven over-installed skills removed (docx, pptx, xlsx, frontend-design, skill-creator, webapp-testing, vercel-react-best-practices, web-design-guidelines, vercel-composition-patterns, vercel-react-native-skills, nextjs-app-router-patterns)
2. Only `find-skills` kept — the lightweight meta-skill, the one that finds the others on demand
3. `skills-sh.md` rewritten from shopping list into a catalogue reference card (14 skills by domain)
4. `athanasios.md` updated: per-project install/remove discipline, explicit context-cost rule
5. Committed and pushed

---

## Film Study

Read this as a design ambiguity that became a Superman failure the moment context cost wasn't weighed. The task surface said "install" and the store made installing trivial — one `npx skills add` per skill — so Superman did the literal thing. But that's exactly the judgement gap. Installing isn't acquiring knowledge; it's loading a payload permanently into context. Knowing the React skill *exists* costs one line in a catalogue card. Having it installed costs its full weight in every session forever, whether the project touches React or not. Superman optimised for capability-on-hand and forgot the team's first principle: lean context is the feature, not a constraint to work around. Entry 2 praised Superman for wiring the routing so the tool survived the session — that instinct was right. The error here was the same instinct overshooting: he tried to make *every* capability survive, when the whole point of `find-skills` is that capabilities don't need to be resident to be reachable. The catalogue is the memory; the install is just-in-time.

Credit where due: the correction was clean and fast. No defensiveness, no half-measure, no "but we might need it." Steve flagged it, eleven skills came out, the skill card was reframed from list to reference, and Athanasios's protocol was hardened so this doesn't recur project-to-project. One push, done. That's the right response to a caught error — the speed of the fix matters as much as the fix itself.

---

## Pattern to flag

This is the second context-discipline lapse worth noting against the squad's stated first principle. The squad reaches for *more* — more skills resident, more rules restated, more capability on standby — when the principle Steve repeats most is *less, deliberately*. The failure mode isn't laziness; it's over-eagerness. The team wants to be ready for everything, and "ready" gets misread as "loaded." It should be read as "reachable." Watch for this graduating: if a third instance of context-bloat-by-good-intention appears, it goes to the library as `context-cost-blindness` — the team treats context as free because it doesn't show up on the cost summary the way tokens-per-call does.

Distilled lesson for Superman's and Athanasios's `lessons.md`: *Installing a skill loads it into context permanently. Default to the catalogue, not the install. Fetch per-project, remove when done. Reachable beats resident.*

---

## Progress Notes

- **Superman:** Two trends running in opposite directions this session. Rule-to-execution gap: still improving (Coach fired again). But a new watch-item — context-cost judgement. He weighs whether a tool survives the session; he doesn't yet weigh what it costs to keep it resident. Not a regression, a blind spot. Trend: improving, with one eye open.
- **Athanasios:** His protocol absorbed the correction well — per-project install/remove is now his default. The scout doesn't hoard. Good.
- **Steve as the gate:** Worth naming — Steve is still catching the things the squad's own principles should catch automatically. The first principle exists in config; it didn't fire as a check before the install. The team has the value written down but not yet wired as a reflex.

---

## Recommended Drill

**Target:** Superman / Athanasios (shared)
**Weakness:** Context-cost blindness — treating "install / load / keep resident" as free
**Drill:** Before adding anything that loads permanently into context (a skill, a standing rule, a resident reference), state the context cost out loud and the trigger for removing it. If it can't name when it leaves, it doesn't come in.
**Pass criteria:** No permanent install lands without a justification that beats "we might need it." Default answer to a capability gap is the catalogue + `find-skills`, not the install.
**Georgiana's delivery:** Frame it as Steve's principle made into a muscle — "you already know lean context is the win; let's make it the first question, not the correction." Lead with how fast the fix was, then make the check pre-emptive.

---

## Film Study

### The SYSTEM.md miss — most important finding of Entry 4

Vera found the Kryptonite qualifier still alive in `squad/SYSTEM.md` line 13: *"before every Squad or Full mode build."* The session spent significant effort propagating "every mode, no exceptions" across agent files, CLAUDE.md, and new-project-checklist.md. SYSTEM.md — the file that declares itself "Immutable. Every agent follows these always. No exceptions." — was not on the checklist.

This is the fourth propagation miss this session. The first three already graduated to a pattern (`rule-propagation-misses-buried-duplicates`). This fourth instance follows the same structure: the prominent occurrence gets fixed, an authority document is treated as already-done and not searched. The pattern is real and the fix (grep the repo for old phrasing before committing) has been logged in two agent lessons files and one pattern file. That the fix didn't self-apply on the session that produced it is telling: knowing the rule and running it as a reflex are not the same thing yet.

Worth noting: Vera caught it. The quality gate is working. The gap is that Vera is doing work that grep should have done before the commit reached her. That's the drill from Entry 2, still not a reflex.

### The decisions folder — an infrastructure miss that compounded

`.squad/decisions/` was called out by SYSTEM.md and CLAUDE.md as mandatory ("Every decision logged in `.squad/decisions/`") but the folder did not exist in this repo. All session decisions — three-tier knowledge graph design, HTML presentation standard, skills lean philosophy — were made verbally and dispersed across config files with no canonical "why we did this" record. Steve had to name this explicitly ("where is that, for example how these presentation decks in HTML are built") before the gap was closed.

This is the infrastructure-creation miss from Entry 1 repeating at a higher level. Entry 1: Coach game-tape folder didn't exist when Coach was first triggered. Entry 4: decisions folder didn't exist despite the rule referencing it directly. Pattern: the squad wires the rule before it creates the infrastructure the rule depends on. The rule is the easy part; the scaffold it requires gets skipped.

Distilled: **rules that reference a folder need that folder to exist on the same commit that adds the rule.** Not the next session. The same commit.

### Subagent drift on superteam-intro.html

During the Vera sign-off pass, `superteam-intro.html` accumulated 118 lines of unstaged CSS changes (expert pill styling on slide 2) that nobody asked for. Origin: one of the subagents drifted during the quality pass. The changes were not committed — Superman left them unstaged and flagged them to Steve. Correct call. But the pattern is worth naming: subagents with write access will occasionally write. A sign-off pass should not have produced presentation changes. The scope contract was not enforced.

This is a supervision gap, not a code gap. Vera's brief should always include explicit scope: "read and assess only — no writes except to your sign-off report." It didn't this time.

### Paulien's memory — value of the late catch

Paulien had not been tracking the backlog across this session. Steve called it directly at the end. Paulien wrote her memory retroactively from the session summary — which worked this time because the summary was available. In a session without a good summary, retroactive reconstruction fails. Paulien's memory write should be wired as a session-close step alongside Coach's game tape, not left as an ad hoc call when Steve notices the gap.
